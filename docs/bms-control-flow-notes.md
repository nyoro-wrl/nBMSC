# BMS制御フロー調査メモ

## 対象命令

- 原典のBM98仕様で確認できる制御フローは、`#RANDOM n`、`#IF n`、`#ENDIF`。
- `#ENDRANDOM` はBM98原典には出てこない。BMS command memoではnanasi由来の拡張として扱われる。
- ただし、ネストした `#RANDOM` では `#ENDRANDOM` が実質的に必要。非ネストでも可読性と実装の明確さのため常に出力・解釈対象にするのが無難。

## 基本仕様

- `#` で始まる行がコマンド行。コマンド名は大文字小文字を区別しない。
- BMSは実行時にコンパイルされるため、コマンド行の順序は自由とされる。
- `#RANDOM n` は `1` から `n` の自然数を生成する。
- `#IF n` から対応する `#ENDIF` までのブロックは、現在スコープの乱数値が `n` のときだけ有効。
- 制御フローは時間的な区間指定ではなく、BMSコマンド行を条件付きで有効/無効にする仕組み。
- 共通部分は `#RANDOM` ブロック外、または少なくとも `#IF` ブロック外のトップレベルに置く。`#RANDOM` ブロック内に `#IF` に属さない共通行を置くのは非推奨で、実装によって無視される。
- 条件一致した `#IF` 内の行とトップレベルの共通行は、展開後の同じBMSに共存する。そのため同じタイミング・同じレーンで交わることは仕様上あり得る。衝突した場合の扱いは、展開後の通常BMSとしての重複・多重行処理に従う。
- `#RANDOM` ブロック外のトップレベル行は常に有効。
- `#RANDOM` ブロック内に `#IF` に含まれない孤立行を置くのは非推奨。BMSEなどでは無視される例がある。
- `#ENDRANDOM` 非対応実装は単に無視する。対応実装では `#RANDOM` ブロックの終端を明示する。
- `#ENDRANDOM` 後に現在スコープの乱数値をどう扱うかは実装差がある。IIDXv/HDXは現在スコープの値を `1` に戻すとされる。

## 拡張命令との関係

- `#ELSEIF`、`#ELSE`、`#SETRANDOM`、`#SWITCH`、`#CASE`、`#SKIP`、`#DEF`、`#ENDSW` なども存在するが、今回の「標準的な4命令」とは別扱い。
- 4命令だけに対応する場合でも、将来拡張しやすいよう、制御フロー評価器はスタック/スコープを持つ形にする。

## BMSE/iBMSC/BmsMaker3系の対応

- BMS command memoのアプリ一覧では、BMSEは「BMx Sequence Editor 1.3.8」、iBMSCは「iBMS BMS Creator 3.0.5 Delta」として記載される。
- BMSEは `#RANDOM` セクションを解析し、Expandタブへ分離できるが、部分対応。
- BMSEのExpandタブは最大65535文字とされ、多数の `#RANDOM` ブロックで溢れる例がある。
- iBMSC 3.0も部分対応。Expansion Codeタブへの分離はできるが、制御フロー表現がEdit Panelにも表示される不完全さがある。
- BMSE v3公式リポジトリでは、`modInput.vb` の `LoadBMSLine` が `#IF` / `#RANDOM` / `#RONDAM` を見つけると `m_blnUnreadFlag = True` とし、行を `m_strEXInfo` に保存する。
- 同じ処理で `#ENDIF` は `m_blnUnreadFlag = False` として `m_strEXInfo` に保存する。`#ENDRANDOM`、`#ELSE`、`#ELSEIF`、`#SETRANDOM`、`#SWITCH` 系の専用処理は確認できなかった。
- BMSE v3の出力では `txtExInfo.Text` をヘッダー・定義の後、MAIN DATA FIELDの前にそのまま書き出す。つまり分岐を「評価」するのではなく、エディタ本体で扱わない追加情報として保存する方針。
- BmsMaker3マニュアルでは、BMSモードが `#RANDOM` などの制御構造に正式対応していると説明される。
- BmsMaker3の制御構造ページでは、BMS command memoを参考に `#RANDOM #SETRANDOM #IF #ELSEIF #ELSE #SWITCH #SETSWITCH #CASE #DEF` を実装し、入れ子も無制限に対応するとされる。
- ただしBmsMaker3でも、ほとんどのBMSプレイヤーは `#RANDOM #SETRANDOM #IF` しか実装しておらず、入れ子にも対応していない点を注意事項としている。
- BmsMaker3は `#ELSEIF` を含むBMSを正常解釈できるが、エディタ上で `#ELSEIF` を追加することはできない。`#IF` で代替できるため不要と判断されている。
- BmsMaker3は分岐枝への「降下」により、その枝の内容を譜面表示エリアと情報ペインへ反映し、親から継承した要素を薄く表示する。
- BmsMaker3は2025-04-13時点のマニュアル記述で非オープンソース。公開GitHubリリースのSource CodeはマニュアルHTMLリソースであり、本体ソースではない。

## nBMSC現状

- `src/Editor/ChartIO.vb` のBMS読み込みは、ヘッダ読み込み時に `#IF` / `#ENDIF` / `#SWITCH` / `#SETSWITCH` / `#ENDSW` を拡張欄へ退避する。
- コメント上は `random` / `setRandom` / `endRandom` が想定されているが、現状コードでは `#RANDOM` / `#ENDRANDOM` 専用処理はない。
- ノート読み込みの2パス目では `xStack` が更新されないため、分岐内のノートを分岐評価せず通常ノートとして読み込む可能性がある。
- 保存時は拡張欄をそのまま出力したうえで、読み込んだノートからMAIN DATA FIELDを再生成する。そのため分岐譜面を読み込んで保存すると、分岐ブロックと再生成ノートが重複・混在するリスクがある。

## nBMSC表示モデル案

- 入れ子構造は初期対応から除外する。
- エディットパネルでは、現状の全分岐重ねがけ表示を避ける。
- 画像編集ソフトのレイヤー構成に近い考え方で、RANDOMブロックを追加・削除できるようにする案がある。
- 共通レイヤーに表示/非表示を持たせる。
- 共通レイヤーが表示の場合、RANDOMブロックの表示結果と重ねて表示する。
- 共通レイヤーが非表示の場合、共通配置を隠し、RANDOMブロック側の配置だけを確認できるようにする。
- 共通レイヤーの表示状態は編集対象の所属とは別に扱う。RANDOMブロック未選択時の新規配置先は、共通レイヤーが非表示でも共通配置とする。
- RANDOMブロックごとに「定義値」「値」「表示状態」を持たせる。
  - 定義値: BMSへ `#RANDOM n` として書かれる値。
  - 値: エディタ上で現在対象にする `#IF n` の値。
  - 表示状態: 全表示、表示、非表示。
- 値は `1..定義値` の範囲を基本とする。
- 値は表示状態とは独立した、現在の編集対象分岐を決める値。
- 表示状態の意味:
  - 全表示: そのRANDOMブロックの全分岐配置を表示する。
  - 表示: そのRANDOMブロックの「値」に対応する分岐配置だけを表示する。
  - 非表示: そのRANDOMブロックの全分岐配置を非表示にする。
- デフォルトの表示状態は「表示」。
- 表示されている分岐配置は編集できる。
- 表示状態が全表示でも、RANDOMブロック選択中の新規配置先は現在の「値」に対応する分岐とする。
- 画像編集ソフトでレイヤーを選択するのと同様に、RANDOMブロックを1つ選択できるようにする。
- RANDOMブロック選択中に新規配置したノートは、選択中RANDOMブロックの現在値に属する分岐配置として扱う。
- 共通レイヤーを編集するため、RANDOMブロック未選択状態へ切り替える手段を用意する。
- RANDOMブロック未選択時に新規配置したノートは、共通配置として扱う。
- 複数RANDOMブロックがある場合、各ブロックの表示状態と値を独立して持つ。
- ノート表示は、共通/分岐の種別そのものではなく、現在の編集主体かどうかで区別する。
- RANDOMブロック未選択時は共通レイヤーを編集主体とする。
- RANDOMブロック選択時は、選択中RANDOMブロックの現在値に対応する分岐を編集主体とする。
- 編集主体のノートは既存どおり表示する。
- 表示されているが編集主体ではないノートは、点線枠で補助表示する。
- 半透明表現は既に不可視ノートで使っているため、レイヤー識別には使わない。
- 仕様上、`#RANDOM` はノートだけでなく譜面内容全体に作用し得る。初期対応では、パネル上に配置・編集できるイベントを構造化対象とし、それ以外は拡張命令テキストとして保持する。
- 初期対応の構造化対象は、通常ノート、BGM、BGA/LAYER/POOR、BPM/STOP/SCROLLイベント、小節長 `#xxx02`。
- 初期対応のテキスト保持対象は、ヘッダー、WAV/BMPなどの定義差、未使用BPM/STOP/SCROLL定義、入れ子やELSE/SWITCH系など。
- `#RANDOM` 内のノート以外の値は、RANDOMブロックごとの拡張命令テキストとして保持すれば、少なくとも保存時の破壊は避けやすい。
- ただし1ブロックにつき1テキストだけでは、どの `#IF n` に属する行かをUI上で扱いにくい。初期案としては、RANDOMブロック内の `#IF n` ごとに未対応コマンド用テキストを持つ。
- RANDOMブロックの「値」を切り替えると、表示する拡張命令テキストボックスの内容も対応する `#IF n` のテキストへ切り替える。
- 拡張命令テキストボックスを編集した場合、現在の「値」に対応する `#IF n` のテキストへ保存する。
- 表示状態が全表示でも、拡張命令テキストボックスの表示・編集対象は現在の「値」に対応する分岐とする。
- そのテキストは保存時に `#IF n` ブロック内へそのまま戻す。読み込み時に解析できない行や、初期対応外のヘッダー/定義などを格納する。
- この方式は「保持」はできるが、選択中分岐のヘッダーや定義をエディタ表示へ反映する「完全解釈」ではない。
- BPM/STOP/SCROLLイベントは、RANDOM分岐内でもパネル上で配置・編集できる対象にする。
- 分岐内BPM/STOP/SCROLLイベントは、既存の通常保存と同様に実値で保持し、保存時に所属スコープごとに `#BPMxx` / `#STOPxx` / `#SCROLLxx` を連番で再割り当てする。
- 所属スコープは、共通、または各RANDOMブロックの各 `#IF n`。
- 元ファイルのBPM/STOP/SCROLL定義ラベル番号は保持しない。保存時に再生成される。
- 分岐内の未使用BPM/STOP/SCROLL定義だけを保持したい場合は、拡張命令テキストに残す。
- 分岐内小節長 `#xxx02` は、各RANDOMブロックの各 `#IF n` ごとの小節長オーバーライドとして保持する。
- エディットパネルの小節長グリッドは、現在選択中のレイヤーを基準にする。Base選択時はBase小節長、RANDOM選択時は選択中 `#RANDOM/#IF` の有効小節長を使う。
- レイヤー切替時は、ノートの始点と終点を「小節番号 + 小節内比率」を保ったまま現在の小節長マップへ変換する。
- `Visible All` 時も小節長グリッドは選択中RANDOMブロックの現在 `#IF` 値を基準にし、他のIFレイヤーのノートもそのグリッド上へ変換表示する。
- 初期対応では、RANDOM分岐内 `#xxx02` を読み込み・表示・保存対象とするが、UIからの分岐小節長編集は行わない。RANDOM分岐選択中は既存の小節長編集操作を無効にする。

## 構造化対象外としてテキスト保持するケース

- 以下を含むRANDOMブロックまたは分岐は、初期対応では構造化編集対象外とし、含まれるノートもパネル上のNoteとして扱わず、該当分岐の拡張命令テキストに格納する。
  - 入れ子の `#RANDOM` / `#IF`
  - `#ELSEIF`、`#ELSE`、`#SETRANDOM`、`#SWITCH`、`#SETSWITCH`、`#CASE`、`#DEF`、`#SKIP`、`#ENDSW`
  - `#RANDOM` ブロック内で `#IF` に属さない行
- 分岐ごとの `#WAVxx` / `#BMPxx` などの定義差は、該当分岐の拡張命令テキストに格納する。
- 分岐ごとにヘッダーや定義が変わる場合も、該当分岐の拡張命令テキストとして保存し、エディタのヘッダー欄や定義リストには反映しない。
- 全表示中に複数分岐のノートが同じ位置へ重なる場合は、そのまま重ねて表示する。どれを前面/背面にするかは初期対応では管理しない。

## 追加仕様

- `#ENDRANDOM` が省略されたBM98原典型の記述は、連続する `#IF` 群までを1つのRANDOMブロックとみなす。
- 分岐内のノート行と未対応コマンド行の元の相対順序は、Note配列とExtraTextへ分離すると完全には保持できない。必要なら分岐内の元行順を保持する構造を追加する。
